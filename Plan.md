## 1. 핵심 판단

1. **PDF 텍스트 추출이 최대 리스크다.** 이 PDF는 폰트 인코딩이 특수해서 pypdf 계열 추출 시 한글이 깨진다(검증 완료). 추출은 비전 기반으로 간다. LLM 에이전트가 Read 도구로 페이지를 직접 읽는다(요청당 최대 20페이지).
2. **정확성의 기준점(ground truth)은 언제나 PDF 원본이다.** 추출 노트는 중간 산출물일 뿐 검증의 기준이 될 수 없다. 사실 검증은 반드시 원본 페이지 재열람으로 한다(P2-4). 비전 오독이 노트를 거쳐 문서까지 전파되는 순환 검증을 구조적으로 차단한다.
3. **개념 품질이 시각화보다 우선이다.** 콘텐츠 파이프라인(Phase 1~2)에 전체 노력의 60%를 배분한다.
4. **데이터와 뷰를 분리하고, 파생 데이터는 전부 빌드가 만든다.** 정본은 `data/concepts.json` 하나. `tools/build.mjs`가 검증, 좌표, edges 파생, searchText, OG 스텁, 배포 복사를 일괄 수행한다. 번들러는 없지만 **데이터 빌드 스텝 1개는 있다**. 사람과 에이전트는 파생 필드를 손으로 쓰지 않는다(이중 저장의 어긋남을 원천 차단).
5. **첫 30초와 공유 장면도 설계 대상이다.** 로딩, 온보딩, 투어, OG 미리보기는 장식이 아니라 이 위키가 "예쁜 점구름"으로 끝나는지 "읽는 은하"가 되는지를 가르는 요구사항이다(6.4절).
6. **기능과 게이트는 대칭이어야 한다.** 6절에 쓴 모든 사용자 대면 기능은 7절에 대응 검사가 있어야 한다. 게이트에 없는 기능은 조용히 증발한다.

## 3. 아키텍처: 4단계 파이프라인

```
Phase 1 추출              Phase 2 개념화               Phase 3 구축               Phase 4 검증
─────────────            ──────────────────         ─────────────────         ─────────────
페이지 매핑+가설 검증       계층 큐레이션(장 재편성 포함)   레이아웃 빌드(좌표/스텁 굽기)   데이터 게이트(자동)
파일럿(표 포함, 점검 0)     위키 집필(입력 소스 고정)      Three.js 은하(모델 확정)      원본 재대조(페이지 단위)
이중 읽기 캘리브레이션 판정  related 정합(edges는 파생)    로딩/투어/온보딩/접근성        UX 게이트(측정 기준 명시)
→ 통과 후 병렬 본대        검수(원본 재대조, 치명 0건)     공유 스텁/딥링크/모바일        GOLD 행동 로그 + RED 채점
```

각 Phase는 산출물 파일이 다음 Phase의 유일한 입력이다. 에이전트 간 통신은 `data/` 아래 파일로만 한다.

## 4. Phase 1. 추출 (비전 기반, 병렬)

### P1-0 페이지 매핑과 가설 검증
에이전트 1명이 PDF를 훑어 산출한다: `data/page_map.json`(장별 PDF 페이지 범위, 책 페이지 환산표, 표지/간지/화보 등 예외 페이지 목록), 가설 A/B 검증 결과. 가설이 하나라도 틀리면 멈추고 사용자에게 보고한다. **P1-1의 읽기 공통 지시는 이 검증 결과로 최종 확정한다**(미검증 가설을 프롬프트에 하드코딩하지 않는다).

### P1-P 파일럿 (점검 0, 본대 투입 전 필수)
추출 에이전트 1명이 10페이지를 아래 스키마로 추출한다. **표가 포함된 페이지 1장을 의무 포함**한다(부록 스타트업 리스트 대비). 오케스트레이터가 (a) 스키마 준수, (b) 판독 품질(원본 페이지 직접 대조)을 확인한다. 미통과 상태에서 본대 투입 금지(11절). 173페이지 비전 읽기는 재실행 비용이 가장 큰 구간이므로 프롬프트 결함은 여기서만 싸게 잡을 수 있다.

### P1-C 이중 읽기 캘리브레이션 (본대 투입 전 판정 완료)
전체 페이지의 5%를 서로 다른 에이전트 2명이 독립적으로 읽는다. **측정 절차**: 제3의 판정 에이전트 1명이 양쪽 노트에서 `kind: 수치|고유명사`인 facts만 추출해 (bookPage, 정규화된 값) 쌍의 집합을 비교한다(수치는 단위 정규화, 고유명사는 aliases 정규화). 불일치율 = (한쪽에만 있거나 값이 다른 쌍) ÷ (합집합 쌍 수). 결과는 `data/calibration_report.json`. **3% 초과 시 중단**하고 추출 전략을 재협의한다.

### P1-1 병렬 추출
- **배정과 산출물 단위.** 부하 기준은 책 페이지, 1인당 책 15~20페이지(= PDF 8~10페이지), 약 18~22명 병렬. 담당 경계는 PDF 1페이지씩 겹쳐 배정(오버랩). **산출물 파일은 에이전트 단위다**: `data/extracted/agent01.json` ~ (담당 pdfPages와 걸친 장 번호 명시). 장 단위 재편성과 경계 병합(페이지, 장 모두)은 부 단위 중간 큐레이터가 수행한다(장이 두 에이전트에 걸릴 때의 소유권 충돌을 원천 제거).
- **읽기 지시(공통 프롬프트, P1-0 결과로 확정).** "PDF 1페이지에는 책 2쪽이 좌/우로 있다(확정된 조판 기준). 왼쪽 쪽을 위에서 아래로 전부 읽은 뒤 오른쪽으로 넘어간다. 각 쪽의 책 페이지 번호를 기록한다. page_map의 예외 페이지는 이 지시를 적용하지 않고 페이지 성격만 기록한다. 표는 문장으로 풀지 말고 tables에 행/열 구조 그대로 옮긴다. 읽히지 않는 부분은 추측하지 말고 갭으로 기록한다."
- **출력 스키마(형식 고정).**

```json
{
  "agent": 1,
  "pdfPages": [17, 26],
  "chaptersSpanned": [1, 2],
  "chapters": [
    {
      "chapter": 1,
      "bookPages": [17, 33],
      "summary": "장 요약 300자 내외",
      "facts": [
        { "text": "원문 그대로의 사실", "bookPage": 18, "kind": "수치|고유명사|날짜|인용" }
      ],
      "tables": [
        { "bookPage": 310, "title": "표 제목", "columns": ["..."], "rows": [["..."]] }
      ],
      "candidates": [
        {
          "name": "프롬프트페이",
          "aliases": ["PromptPay"],
          "definition": "한 줄 정의",
          "bookPages": [80, 85],
          "evidence": { "mentions": 4, "inChapterTitle": false, "chaptersSpanned": 1 }
        }
      ]
    }
  ],
  "gaps": [{ "pdfPage": 45, "reason": "이미지 해상도" }]
}
```

  중요도 점수는 추출 단계에서 매기지 않는다. `evidence`만 기록하고 최종 1~5 부여는 최종 큐레이터가 전역 기준으로 일괄 수행한다(에이전트별 관대함 편차로 별 크기가 왜곡되는 것을 차단).
- **판독 불량 표준 폴백.** 갭 페이지 재추출은 `pdftoppm -r 200 -f N -l N -png`으로 렌더한 뒤 좌/우 반쪽을 crop해 책 1쪽 = 이미지 1장으로 Read한다(2단 조판 순서 문제도 물리적으로 해소). 캘리브레이션 실패 시 이 경로를 전체 전략으로 승격할 수 있다.
- **실패 처리.** 갭은 `data/extraction_gaps.json`에 집계. 전체의 10% 초과 시 중단, 사용자 보고.

## 5. Phase 2. 개념화 (이 프로젝트의 심장)

### P2-1 계층 큐레이션 (컨텍스트 초과 방지)
1. **부 단위 중간 큐레이터 6명**(1~5부 + 부록): 담당 agent JSON들을 장 단위(`data/extracted/ch01.json` ~)로 재편성하고 경계 병합, candidates 통합, 중복 제거, 표기 정규화(aliases 병합), 부 내 상위 후보 목록 생성. 부별 소개문 `galaxies[].intro` 1~2문장을 장 요약에서 발췌해 작성한다(지어내지 않는다).
2. **최종 큐레이터 1명**: 6개의 요약된 후보 목록만 받아 전역 통합. 유형 분류, 중요도 1~5 일괄 부여, 200개 이상 확정, **투어 코스 선정**(`meta.tour`, 부별 균형을 맞춘 순서 있는 id 15~20개).

**중요도 분포 제약**(라벨 정책과 UI 전제의 붕괴 방지): 5등급 ≤ 10개, 4등급 이상 합계 ≤ 25개. 게이트에서도 검사한다.

모든 병목 에이전트(중간/최종 큐레이터, 그래프 빌더 포함)는 **완전성 자가 확인**을 의무화한다: "입력 파일 개수와 항목 수를 먼저 세어 보고하고, 기대치와 다르면 진행하지 말고 보고하라."

확정 기준: 유형 배분 가이드(기술/서비스 30%, 기업/기관 25%, 정책/제도 15%, 현상/트렌드 15%, 인물 5%, 용어 10%, 파일럿 결과에 따라 조정 가능), 여러 장에 걸친 개념 우선. 산출물 `data/concept_list.json`.

### P2-2 위키 집필 (병렬)
집필 에이전트 12~15명, 1인당 15~20개. **입력은 (a) concept_list.json의 담당 항목, (b) 해당 개념의 근거 장 extracted JSON 전문**, 두 가지로 고정한다. 각 문서는:
- summary 1~2문장, body 300~600자(마크다운), **원문 인용 1개**(`quote: { text, bookPage }`)
- **quote는 extracted JSON의 facts 중 kind가 '인용'인 항목 또는 원문 그대로 기록된 text에서만** 가져온다. 기억으로 재구성한 인용은 치명 오류로 분류한다
- 책의 내용만 근거로 쓴다. 책에 없는 수치, URL, 성과를 지어내지 않는다
- `bookPages` 명시, related 후보 3개 이상 제안. related는 **concept_list의 id만** 쓸 수 있고, 목록 밖 제안은 `proposedNew`에 따로 적는다(큐레이터가 판단). 임의 id 생성 금지

### P2-3 관계 정합
그래프 빌더 1명이 `related`를 정합화한다. **edges는 저장하지 않는다. build.mjs가 related에서 파생 생성한다**(cross는 galaxy 비교로 자동 산출). 이중 저장의 어긋남을 원천 제거.
- 양방향 정합(A→B면 B→A), 고아 노드 0개, **노드당 링크 상한 8개**(쏠림과 칩 스크롤 지옥 방지)
- 쳐낸 링크로 평균이 3 미만이 되면 해당 개념을 집필자에게 반려해 보강한다. **그래프 사유 반려도 문서당 반려 횟수에 합산**하며, 2회 초과 시 오케스트레이터가 인접 개념과의 통폐합 여부를 판단한다

### P2-4 콘텐츠 검수 (원본 기준)
- **RED 사실 검증. 기준점은 PDF 원본이다.** 재대조는 문서 단위가 아니라 **페이지 단위로 최적화**한다: 같은 bookPages를 근거로 하는 문서들을 묶어 해당 원본 페이지를 1회만 재열람하고 그 페이지 근거 문서 전체를 함께 대조한다. 대상: 수치/고유명사 포함 문서 전수 + 무작위 10%. **quote는 전수 검사**(해당 bookPage에 실제 존재해야 하며 불일치는 치명). 예상 재열람 페이지 수는 P2-1 완료 시점에 집계해 점검 2에서 보고한다.
- **불합격 등급화**: 치명(수치/고유명사 오류, 지어낸 내용, 재구성 인용) / 경미(표현). 통과 기준 **치명 0건**. 샘플에서 치명 1건 발견 시 해당 집필자 배정분 전수 재검(에스컬레이션).
- **BLUE 가독성 검수**: 책을 안 읽은 독자에게 읽히는지.
- **재작성 루프 상한**: 문서당 반려 2회(그래프 사유 포함). 3회째는 오케스트레이터가 원인을 판단한다.

최종 산출물 **`data/concepts.json`** (8절).

## 6. Phase 3. 시각화 구축

### 6.1 기술 결정 (확정 사항, 프론트엔드 에이전트의 재량 아님)

- **스택**: Three.js **버전 1개 고정(r170)**. 확보 절차: `npm install three@0.170.0` 후 `build/three.module.js`, `examples/jsm/controls/OrbitControls.js`, `examples/jsm/renderers/CSS2DRenderer.js`를 vendor로 복사, 파일 해시를 README에 기록. 바닐라 JS ES 모듈, 번들러 없음.
- **import map 필수.** addons는 내부에서 `import 'three'`(bare specifier)를 하므로 vendor 방식에서는 import map 없이는 100% 깨진다:

```html
<script type="importmap">
{ "imports": { "three": "./vendor/three.module.js",
               "three/addons/": "./vendor/addons/" } }
</script>
```

- **경로는 전부 상대경로.** GitHub Pages 서브패스(`/저장소명/`)에서 절대경로는 깨진다. 예외는 OG 태그(6.4, 절대 URL 필요)로, `__BASE_URL__` 플레이스홀더 + README의 배포 전 치환 절차로 처리한다.
- **별 본체는 InstancedMesh 확정.** 200~300개 규모에서 노드별 크기/색/정밀 클릭이 필요하므로 Points가 아니라 InstancedMesh다(Points는 레이캐스트 threshold가 월드 고정값이라 줌 레벨에 따라 오클릭). `InstancedMesh(IcosahedronGeometry(1,1), material, N)` + 인스턴스별 스케일 + `instanceColor`. 클릭 판정은 `intersection.instanceId → concept id 맵`.
- **글로우는 단일 THREE.Points 1개**(별과 동일 좌표의 버텍스, 방사형 그라데이션 텍스처, AdditiveBlending, 버텍스 셰이더에서 `gl_PointSize = size·importance/거리`). **드로우콜 1.** 글로우는 레이캐스트 대상에서 제외한다(클릭은 본체만). 별당 Sprite 300개는 드로우콜 300개라 금지. UnrealBloomPass는 데스크톱 한정 옵션.
- **모바일**: `renderer.setPixelRatio(Math.min(devicePixelRatio, 2))`.
- **라벨 정책**: importance ≥ 4는 CSS2DRenderer 상시 라벨(분포 제약상 최대 25개), 나머지는 호버 시 1개만. 호버 레이캐스트는 pointermove마다가 아니라 **rAF당 1회**, 마지막 포인터 좌표로.
- **마크다운 렌더**: bold/리스트/인용만 지원하는 자체 30줄 변환기.

### 6.2 레이아웃: 좌표는 빌드 타임에 굽는다

**은하 모델 확정: 태양 중심 단일 은하 + 부별 나선팔 5개.** (분리 성단 5개 모델은 기각: 은하군 간 3D 겹침 검사가 추가로 필요하고, 부 구분은 팔 색만으로 충분하다.) 은하 평면은 하나, 팔별 y 지터로 입체감을 준다.

`tools/layout.mjs`가 결정론적으로 좌표를 계산해 `position: [x,y,z]`에 저장한다. 런타임 무작위 배치 금지.

```
// 팔 g (1~5부): 오프셋 θg = g·2π/5
// 개념 i (importance 내림차순, 큰 별부터 안쪽):
t = i / N_g
r = R_inner + t·(R_outer − R_inner)
θ = θg + b·ln(r/R_inner) + jitter(seed,i)·0.3        // 로그 나선
p = [r·cosθ, jitterY(seed,i)·H, r·sinθ]
// 겹침 방지: dist < (s_i + s_j + margin)이면 jitter 재샘플 최대 10회, 실패 시 r += margin
// 부록: 도넛 분포 r ∈ [R_outer+4, R_outer+8], θ 균등, y 소폭 지터 (소행성대)

// 초기 파라미터 (SVG 프리뷰에서 조정, 초기값 없이는 프리뷰를 못 돌린다):
R_inner=8, R_outer=40, b=0.35, H=3, margin=1.2
s(importance) = 0.5 + 0.3·importance
```

시드 고정(mulberry32, `meta.seed`). 레이아웃 결과는 **2D 평면 산점도 SVG 프리뷰**로 뽑아 Three.js 착수 전에 승인받는다(점검 3-a). 배치 재작업이 가장 싼 지점이 여기다.

`tools/build.mjs`의 일괄 작업: (a) 스키마 검증, (b) layout 호출, (c) `searchText` 생성(**name + aliases + summary + body** 마크다운 스트립), (d) **edges 파생 생성**(related에서, cross 자동 산출), (e) **개념별 OG 스텁 생성** `docs/c/{id}.html`(og:title = 개념명, og:description = summary, og:image 절대 URL은 `__BASE_URL__` 치환, 로드 즉시 `../#id`로 이동), (f) `docs/data/concepts.json` 출력. 정본 이원화는 게이트의 해시 일치 검사로 차단한다.

### 6.3 씬과 카메라

- 태양: 중앙 발광 구체. 클릭 시 책 소개 패널(저자의 한 문단 "이 책을 은하로 만든 이유" 자리, 책 구매/소개 링크 자리. 내용은 사용자가 채운다, 지어내지 않는다)
- **공전은 별 개별 이동이 아니라 은하 전체 Group의 느린 `rotation.y`(주기 240초)로만 구현한다.** 인스턴스 로컬 매트릭스 불변, 레이캐스트는 Group 변환 자동 반영이라 추가 비용 없음. **패널 오픈 중에는 Group 회전을 일시정지**한다(카메라 비행 목표가 움직이는 표적이 되는 문제, 딥링크 복원 위치 문제를 3줄로 해소).
- 초기 카메라: `position = (0, 0.6R, 1.8R)`(R = R_outer), fov 55, target 원점
- 별 클릭 시 카메라 비행:

```
end = P + normalize(camera.position − P) · max(6·s, minDist)   // near 클리핑 방지
매 프레임 k = easeInOutCubic(elapsed/800ms):
  camera.position = lerp(start, end, k); controls.target = lerp(t0, P, k)
비행 중 controls.enabled = false, 완료 시 true
비행 중 새 목표 클릭 시: 현재 트윈 즉시 파기, 현재 위치에서 새 트윈 시작 (대기열 없음)
```

- 관계선: 기본 숨김, 별 선택 시 그 별의 edges만 발광 표시. cross 링크는 색을 달리한다.

### 6.4 경험 설계 (첫 30초, 탐색, 공유, 접근성)

- **로딩 시퀀스**: 태양 즉시 렌더 + "별을 만드는 중… 132/200" 카운터 → 별이 퍼져나가는 3~4초 인트로 카메라. 검은 화면 금지. **건너뛰기 버튼 상시 표시, 재방문(localStorage) 또는 해시 딥링크 진입 시 인트로 생략**하고 즉시 해당 별로 비행.
- **첫 방문 코치마크 3개**: "드래그로 회전, 휠로 확대, 별을 클릭". 첫 방문에만, localStorage로 재노출 방지.
- **시작 안내**: importance 5 별 3~5개에 은은한 펄스 + "여기서 시작하세요".
- **투어 모드**: 상단 "투어" 버튼. `meta.tour`의 별 15~20개를 1부→5부 순서로 자동 순회. **부 전환 시 `galaxies[].intro` 소개 카드 1~2문장 표시, 진행 표시 "7/18", 일시정지/다음/종료 컨트롤.** 책의 목차가 곧 여정이 된다.
- **읽은 별 점등**: 방문 개념을 localStorage에 기록, 별 밝기 변경 + "200개 중 37개" 진행 표시. (선택 사항: 한 은하팔 완주 시 해당 팔이 한 번 반짝이는 마이크로 피드백. 9.5 게이트 필수 항목은 아니다.)
- **패널 (데스크톱)**: 우측 슬라이드 패널. 개념명, 유형 배지, 부/장(장 제목은 `chapters` 테이블 조회), 원문 인용 블록(bookPage 표기), 본문, 유관 개념 칩(클릭 시 비행), **"다음 별 추천 1개"**(결정론 규칙: related 중 미방문 우선, 동률이면 importance 높은 것. "다음: 트루머니 →" 버튼), **하단 고정: "이 내용은 책 N~M페이지에 있습니다 · 책 소개 보기(링크 자리, 사용자가 채움)"**, "이 별 공유하기" 버튼.
- **패널 (모바일)**: 바텀시트(요약 살짝 → 위로 끌면 전문). 3D를 가리지 않는다.
- **공유와 딥링크**: `#promptpay` 해시 접속 시 그 별로 비행 + 패널 자동 오픈(상태 복원). 공유 버튼은 **개념별 OG 스텁 URL**(`c/{id}.html`)을 복사한다. 해시 자체는 크롤러에게 안 보이므로 개념별 미리보기는 스텁이 담당한다(6.2-e). 사이트 대표 OG는 index.html에.
- **접근성 (3D 비의존 경로 포함)**: (a) 검색 입력에서 방향키로 결과 이동, Enter로 별 선택. (b) 패널은 시맨틱 HTML + aria-label, Esc로 닫기. (c) **"목록 보기" 토글**: 은하팔별 개념 전체 인덱스, 클릭 시 동일 패널. 마우스 없는 사용자, 스크린리더 사용자, 저사양 기기의 최종 폴백(6.5)을 겸한다.
- **멀미 대응**: 모션 정지 토글 버튼, `prefers-reduced-motion` 감지 시 자동 정적 모드, 카메라 비행은 완만한 이징 고정. 정적 모드에서는 시작 안내 펄스도 모션이므로 끄고, 정지된 하이라이트 링으로 대체한다.
- **검색/필터**: 검색 대상은 searchText(aliases 포함이라 "PromptPay"로도 걸린다). **결과 0건 시 "가까운 별" 3개 제안**(필터 무시한 부분 일치). 은하팔 필터, 유형 필터.

### 6.5 성능 폴백 (단계적 하향)

- **감지 로직**: 직전 120프레임 이동 평균 rAF 간격 > 33ms가 **2초 연속** 유지될 때 발동(순간 드랍, 탭 전환, GC에 반응하지 않는다). 인트로 재생 중에는 감지 유예. 하향은 단방향(복귀 없음).
- 하향 순서: 1단계 DPR 하향 → 2단계 글로우 Points 제거 → 3단계 Group 회전 정지 → 최종 폴백은 "목록 보기"(6.4). (별 개별 자전은 이 설계에 존재하지 않는다. 없는 기능을 구현하지 말 것.) 3단계 후에도 미달일 때만 2D 전환을 사용자와 협의한다.
- fps는 `?debug` 플래그의 rAF 간격 카운터로 측정한다.
